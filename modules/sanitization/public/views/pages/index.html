Input sanitization techniques

This page shows you different results for different techniques of handling potentially dangerous strings.

Potentially dangerous strings are usually user-input, query params, form data, etc. To avoid users injecting harmful code into your application you have to think about input sanitization.

Source code of this example is on Github
It is very important that you compare the code that has been used to generate those results.

`html_safe` filter

Without html_safe

This will just print text in the source code of the page, because it is automatically sanitized.

{{ '' }}

With html_safe

This code will execute, because it is marked as "safe" using html_safe filter. You can verify that by opening your devtools console.

{{ '' | html_safe }}

`print` tag

{% liquid assign safe_script = "" assign malicious_script = "" %} {% capture result %}{{ malicious_script }}{{ safe_script | html_safe }}{% endcapture %}

Without print

This code will not execute the second script, even though it is marked as html_safe. 

{{ result }}

With print

Print tag allows us to mix html_safe and unsafe strings into one, which in result renders first variable and executes the "safe script" and prints out "malicious script". You can verify that by opening your devtools console.

{% print result %}

String and variable rendering

String rendering {% raw %}{{ var }}{% endraw %} output is sanitized by default. So if you assign string to a variable, it will be sanitized when you render it.

{% raw %} {% assign user_name = 'Click Me' %}

{{ user_name }}

{% endraw %} {% assign user_name = 'Click Me' %}

{{ user_name }}
